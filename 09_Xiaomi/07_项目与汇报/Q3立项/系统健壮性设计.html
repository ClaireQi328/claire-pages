EP 系统级别建设目标 · 系统如何更健壮

下面这张 Wave 模块图是一张按排期的功能地图——它回答了"先做什么、后做什么"，但没回答"它怎么站得稳"。而我们自己反复警惕的是"不是继续加功能"。

系统健壮性的真问题：当数据是脏的、上游断连了、来了异常件、规则变了、量暴涨了——这堆模块还能不能稳定产出正确决策，而不是崩、不是漏、不是要人救火。

骨架 ≠ 健壮。健壮 = 让骨架长出「自愈能力」。

这张图是骨架；健壮性是脏数据能挡、断连能补、异常能升级、变化能配置、规模能扛。对应到 B 点框架，健壮性恰恰是两大闭环能真正转起来的前提——没有数据/集成/流程的健壮，①②③ 层的模型与 ④ 层的回写就是空转。

B

五个「不怕」· 健壮性设计原则

5 dimensions
1
不怕脏数据
数据健壮最该先夯，也是 A 点最大的坑
主数据单一可信源（SSOT）
价格白账定点台账必须是唯一主动源；询报价、订单都引用它，不允许各模块各存一份。
口径统一 + 入库校验
品类管理供应商 360字段先过校验再入库，挡住"漏绿 / 口径冲突"。
缺数据降级而非崩溃
成本核算Should-cost 因子缺失时退回"行情价 + 标注待补"，而不是算不出来直接卡死。
数据血缘可追溯
每个价格 / 绩效数字都能回答"它从哪来、谁改的、何时改"。
2
不怕断连
集成健壮多系统对接是最脆的点
幂等
同一需求重复导入、重复下单、重复回写台账，结果不变——防 Agent 自动化下的重复操作。
重试 + 超时告警 + 补偿
对接 MIPEC / SAP / 飞书 / SVC 失败自动重试，超阈值推飞书 Bot，绝不"静默丢失"。
明确状态机
合同 / 订单状态只能合法跃迁（"已签署"不能跳回"待起草"），非法跃迁拦截。
对接结果可回查
每次导入 / 下单留完整执行日志与成败状态，可审计、可重放。
3
不怕异常
流程健壮闭环不能断头
异常件自动识别
样件订单"多供应商 / 无供应商"异常件不能混在常规件里被遗漏——独立标记、独立升级队列。
每个自动环都有人工兜底
Phase 1 半自动（Agent 建议→人确认）就是健壮设计，低风险才放 Phase 2 全自动。
异常必须能回写
偏差关闭 → 沉淀规则 → 下次自动处理，否则同样的异常永远在重复触发。
关键节点超时预警
任务中台定点 / 设变 / 变价超时自动升级，进度一览无余、零延误。
4
不怕变化
扩展健壮规则会一直变，法规更会
配置化 > 硬编码
人员路由（分类-MVPS-人员）、价格规则、合规问卷模板都做成可配置表，业务自己改，不用发版。
CLI 原子化（关键）
Agent / Skill / CLI 分层中，CLI 接口稳定 → Skill 自由重组 → 新 Wave 不动老接口，"加功能不动地基"。
Wave 之间松耦合
合同(W1) 挂了不该拖垮询报价(W2)；模块间通过主数据交互，而非硬依赖。
向后兼容
老报价模板、历史合同结构在新版本下仍可读、可查、可追溯。
5
不怕规模
性能健壮200 单/天会涨到几千单
批处理 + 限流
大批量导入 / 卷积计算走异步批处理，别让批量任务把实时下单挤死。
优先级队列
高价值需求优先处理；样件、紧急补料插队，常规件排队不阻塞。
A+B 卷积可增量
总成价格联动DB 件变更只重算受影响总成件，而非全量重跑。
关键查询可缓存
价格白账、品类主数据等高频读热点缓存，降低响应延迟。
一句话收敛

Wave 图给的是骨架，健壮性让骨架长出自愈能力：脏数据能挡、断连能补、异常能升级、变化能配置、规模能扛。这五条不是上线后再补的"非功能需求"，而是两大闭环能转起来的地基。

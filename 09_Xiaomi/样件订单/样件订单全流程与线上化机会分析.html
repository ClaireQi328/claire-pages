全流程详解（标准 B 端纵向流程）

按角色泳道（申请方 / 订单管理员 / 采购员 / 系统·Agent）纵向排布，白底线框风格；决策点做三路并行分支 → 横向汇合（无交叉箭头）；框内标注执行方式（系统 / 手动 / Agent）。采购员拆为订单管理员与采购员。

📌 流程要点
  • 价格匹配三态：有价格（直接自动下单）/ 无价格（触发定价任务）/ 一品多点（Agent 推荐 + 采购员强制确认）。
  • 自动化执行三策略：A 全自动推送 SAP/SZ · B 半自动（人点确认）· C 人工执行（采购员）。
  • 变更升版：V2 生效后才撤 V1，避免审批期间下游系统订单失效无法收货（见设计文档）。
🔒 两道关键闸门（硬性前置）
  • 闸门一 · 需求锁定：需求进入「价格匹配 / 订单生成」前必须处于锁定状态。未锁定禁止向下推进；锁定后若需修改，须先解锁 → 修改 → 重新锁定并留版本备注。责任人：订单管理员；确认方：需求发起方。
  • 闸门二 · 下单前人工确认：订单推送 SAP/SZ 前，无论全自动（A）/半自动（B）/人工（C）路径，均须由采购员 / 订单管理员人工确认并留痕。确认通过方可执行下单；A 路径亦需确认留痕，不可静默自动。

5

核心痛点（7 项）

基于会议纪要（2026-04-27）与升版设计文档梳理。注：订单聚合非当前需求，已不纳入方案。

P-01

需求字段混乱

依赖备注文本，缺项目名称/车型；米Tab 系统项目编码未映射到采购界面。

P-02

一品多点选择

占比约 10%，需求方常填不准供应商，需在审批环节强制采购员确认。

P-03

台账数据质量

价格为 0 但含软模费的无效记录、同零件号同供应商同价格重复项。

P-04

工厂维度缺失

自动下单未考虑送货地，物流价格按工厂差异，造成成本误差（设计文档风险提示）。

P-05

升版状态复杂

V1/V2 撤批时机不当致下游系统订单失效、无法收货（升版设计文档）。

P-06

自动下单覆盖低

仅约 40% 订单可自动下单，其余因一品多点/一供应商多价格需人工。

P-07

预算与车型脱钩

项目编号仅作预算标识，与采购决策无关，易混淆采购项目与车型项目。

⚠️ 核心瓶颈 自动下单覆盖率仅 ~40%，瓶颈来自台账脏数据（P-03）+ 一品多点/多价格（P-02/P-06）；而升版逻辑缺陷（P-05）直接影响下游履约稳定性。

AI 识别点（7 个触发点）

流程中可被系统/ Agent 自动识别并触发的节点，是 Agent 化的「眼睛」。

识别点 01

需求接入触发

多源需求到达，Agent 自动归集到对应项目/车型。

识别点 02

字段补全触发

缺项目/车型时，调用米Tab 映射补全，减少人工追问。

识别点 03

价格匹配触发

有价/无价/一品多点三态分流，路由到不同处理。

识别点 04

供应商推荐

一品多点时 Agent 荐供应商并附 DRE 信息，待采购员确认。

识别点 05

下单触发

规则命中（有价、信息完整）即自动下单推送 SAP/SZ。

识别点 06

升版检测

V2 启动后 V1 保持生效，V2 供应商确认后才撤 V1。

识别点 07

进度查询

自然语言解析查询意图，返回订单状态/物流进度。

🔑 逻辑主线 识别点沿「接入 → 补全 → 匹配 → 推荐 → 下单 → 升版 → 查询」串成闭环，前 2 个解决数据质量，中间 3 个解决匹配与执行，后 2 个解决稳定与可观测。

落地建议

分阶段推进：先数据底座，再自动化，最后智能追踪。

📌 优先级
P0 · 台账清洗 + 字段映射 + 三态规则 P1 · Agent 确认 + 自动下单 + 一品多点强制选 P2 · 升版稳定 + 智能进度追踪 + 自然语言查询
🛠 执行步骤拆解 按 P0→P2 三阶段，将落地动作拆为可执行步骤、责任人与验收标准（订单聚合已按反馈剔除）。
优先级具体执行步骤责任角色验收标准
P0台账数据清洗：剔除价格为 0 但含软模费的无效记录，合并同零件号 + 同供应商 + 同价格重复项。订单管理员 · CLI无效/重复记录清零；自动下单覆盖基线可计量。
P0字段自动映射补全：米Tab 项目编码/车型映射至采购界面，替代备注文本。订单管理员 · Skill字段缺失率下降；录入不再依赖人工追问。
P0价格三态规则引擎：配置有价 / 无价 / 一品多点分流规则。订单管理员 · 规则引擎三态识别准确率 ≥ 95%。
P1Agent 确认 + 自动下单：规则命中自动推 SAP/SZ，复杂转人工。订单管理员 · Agent自动下单覆盖率 40% → 70%。
P1一品多点强制选：审批嵌强制选供应商组件，Agent 推荐 + 采购员确认。采购员 · CLI~10% 一品多点 100% 经强制确认。
P1工厂维度物流价补全：下单补送货地，按工厂区分物流价。订单管理员 · CLI物流成本误差归零。
P2升版稳定：V2 生效后才撤 V1，避免下游失效。订单管理员 · 系统升版期间 0 下游订单失效。
P2智能进度追踪：看板 + Agent 进度汇总 + 自然语言查询。系统 · AgentExcel 手工统计归零。
✅ 关键结论
  • 当前下单自动化率约 40%，经台账清洗 + 规则引擎 + Agent 确认后目标 80%，释放 1 FTE。
  • 改造不是完全替代人，而是把「规则明确、可固化」的环节交给 Agent/CLI，把「判断、异常、强选」留给人工确认。
  • 升版逻辑需先稳住（V2 生效后才撤 V1），避免自动化推进时拖累下游履约。

订单状态机与变更 / 取消规则

从「需求创建」到「完成关闭」共 8 个主状态，其中 需求锁定 与 下单确认 为两道人工闸门（详见 ③）。任一状态下均可发起修改或取消，但允许度随状态递进而收紧：越靠后越不可直接改动，需走升版 / 退货 / 退款等变更流程。

状态推进
取消（红虚线）
变更 / 升版（橙虚线）
橙框 = 人工闸门

各状态下的「修改 / 取消」处理规则

状态可修改修改怎么做可取消取消怎么做关键约束
需求草稿可自由编辑字段、增减零件可直接作废，记录原因未进入系统流程，无成本影响
需求锁定需解锁解锁 → 修改 → 重锁 + 版本备注可解锁后取消，记录原因 + 审批锁定是后续阶段的前置闸门
价格匹配可改字段 → 回匹配重算三态可终止匹配 → 取消一品多点需重新强制选供应商
待下单确认可退回 S2 改价 / 改供应商可终止 → 取消（需说明）下单前人工确认闸门
已下单 / 已确认不可直接改走升版 V2（V1 保持生效至 V2 确认）生成变更单需审批撤单：采购员+订单管理员审批，撤回 SAP/SZ 推送，释放采购承诺已产生采购承诺，取消需审批
已发货不可改内容仅换货 / 补发（变更单）不可直接取消拒收 / 退货流程 → 退货中 → 财务退款物流在途，取消 = 退货
已收货不可改仅售后 / 退货退款（变更单）不可直接取消退货退款流程（质量/错发）→ 财务已进入结算，取消 = 退票退款
完成关闭不可—不可—终态，归档
💡 设计原则
  • 越早越自由：草稿 / 锁定态可自由改与取消，无成本影响；进入下单后任何改动都需走正式变更流程并留痕。
  • 已确认不可直接改：已下单（已确认）走升版 V2，V1 保持生效至 V2 供应商确认，避免下游失效。
  • 已发货 / 已收货不可取消：只能走退货 / 退款流程，由质量或商务发起，财务介入结算。

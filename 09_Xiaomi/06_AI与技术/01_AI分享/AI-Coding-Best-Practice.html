Part 1 · Boundaries

AI Coding 的能力边界

👤 团队负责人 — 提供建议的"军师"，只对结果负责
📋 下属员工 — 需要清晰理解任务，有能力以文字描述要求
🤖 Agent — 可长时间执行任务并交付可用结果，但需要完整背景信息和清晰指令

✅ Agent 已有的能力

  • 已经有自我更新的元能力
  • 可以通过 Tool 和 Skill 扩展，潜力巨大
  • 有丰富的工具可用 (web, image)

⚠️ Agent 当前局限

  • LLM 本身没有记忆（无状态请求）
  • 有效上下文窗口依然是瓶颈
  • 只有"大脑"，缺乏了解真实世界的工具
  • 不能保证交付，结果不一定正确
  • 不对最终交付负责
Part 1 · Pain Points

Vibe Coding 中经常遇到的问题

  • 即使提示词很完善，Agent 的执行计划依然没有达到预期
  • 整理提示词需要花费的精力很大，甚至超过 AI 带来的提效
  • 长时间 Vibe Coding 之后，提示词散落得乱七八糟
  • 如何有效地把企业内部的知识引入到上下文中
  • Agent 循环中改了太多代码，校验有问题，没有排查思路
  • AI Coding 产生了太多代码，如何高效可靠地验证 AI 的产出
  • AI Coding 工具层出不穷且更新很快，怎么才能不掉队
🔑 基本思路：对待 AI 的方式应该像对待新入职的下属
① 了解模型的能力边界 → ② 使用 AI 擅长的方式沟通 → ③ 必要时接管代码 → ④ 有充足技术储备验证产出 → ⑤ 充分利用 AI 自身能力

Part 2 · Agents

Multiple Agent & Subagent

🤝 多 Agent 协同

  • 支持自定义 Agent
  • 通过提示词或模版调度多个 Agent
  • 多 Agent 之间上下文共享
  • 支持 Handoff
  • 可以让某个 Agent 调度 Subagent

示例: github.com/ShepAlderson/copilot-orchestra

🔧 Subagent

  • 执行时是独立上下文
  • 结果汇入主对话上下文
  • 是 tool 之一，支持显式唤起或模型自动唤起
  • 可以多个 Subagent 异步执行
  • 但会阻塞主对话（作为 tool call）
Part 2 · Cost

代码生成成本很低，
但是验证成本很高

🏆 使用最好的模型

  • 提高任务成功率
  • 降低代码检查成本
  • 降低返工成本
  • 不同任务选择不同模型

⚡ 快速验证原型

  • 快速实现 MVP
  • 验证 AI 实现方向
  • 及时通过上下文修正方向
  • 暴露 AI 实现的风险

🔄 让 AI 相互竞争

  • 同一任务交给不同模型
  • 一个模型写代码，另一个检查修正

💰 校验成本很昂贵

  • 让 AI 给出实施步骤并外化
  • 每个任务都包含校验准则
  • Test 先行 (TDD)
  • 警惕 AI 的"奖励劫持"
Part 2 · Decomposition

复杂任务的合理分解

📏 合理规划任务大小

  • 产出代码适合一次 PR
  • 适合人高效 Code Review
  • 任务运行时长在合理区间
  • 不指定超出模型能力的任务
  • 尽量减少任务间耦合

🔀 DAG 任务编排

  • 合理组织先后关系以实现并行
  • 利用 AI 协助做任务分解
  • 推荐 git worktree 做代码版本管理
  • 善用并行工具 (Tmux)

✅ 必须包含测试任务

  • 单元测试必须 100% 覆盖
  • 需要完善的测试报告
  • 验证测试结果也很重要
  • 需要构建自动化测试平台
Part 2 · Security

在安全前提下最大化 AI 能力

🔓 AI 需要放权

  • 工具是 AI Coding 的手脚
  • Bash 脚本权限
  • 操作浏览器
  • 网络权限
  • 文件操作权限
  • 目的是让 AI 可以长时间执行任务

🛡️ 安全的方式

  • DevContainer 最佳
  • 本机使用时需慎重放权
  • 需要有 AI Coding 时代的安全基线

🔐 合理保管密钥

  • Key 和密钥不要让 AI 接触到
  • 尽量利用 Entra ID 做服务认证

🔍 审核 AI 的插件

  • MCP Server 启用认证
  • Skills 要检查描述和脚本
  • 长期记忆里增加安全门禁
Part 2 · Quality Gates

质量门禁是规模化的前提

🛡️ 预防大于补救

  • 验收标准比任务需求更重要
  • 事后补救成本远大于事先预防
  • 规范化 AI 任务验收文档
  • 自动化 AI 驱动的质量门禁
  • 在组织维度实施 SDLC

🧪 TDD

  • 测试分级 (L1-L5)
  • 测试代码先行
  • 利用 AI 完善测试案例和边界
  • 利用 AI 对测试案例做交叉验证

🏗️ 自动化测试体系

  • 需要自动化测试体系做最终验证
  • 慎重对待测试报告
  • 报告需包含数据来源和测试逻辑
  • 利用 AI 对测试报告做交叉验证

🖥️ 为 AI 准备环境

  • 让 AI 看到应用 UI、日志、指标
  • 提供更多 QA 相关的 AI 友好工具

Part 3 · Tips

其他 Tips

⏱️ 长时间运行的任务

  • Agent 运行时间长 → 减少"进程切换"
  • 远程开发是很好的长时间运行方式
  • 长时间运行意味着可以并行跑多个任务

📂 提示词模版管理

  • 重点是用而不是收集
  • 没有通用模版，只有最合适的
  • 理解原理非常非常重要

📋 Plan > 执行

  • [Plan > 执行] 应对大多数场景
  • 慎用 SDD — 前期投入非常高

🔧 推荐工具

  • Tmux：异步并行执行多任务
  • Memorybank：上下文外化的示例

💡 核心指令文件 (instructions.md) 非常重要

  • 记录 Agent 行为准则、项目上犯过的方法论错误
  • 理解 Skill 核心思想：渐进式披露 + 一切上下文都是文件
  • 个人产出天花板 = 并行处理时的上下文切换成本
  • 团队产出天花板 = SDLC 建设能否支撑 AI 的代码生成速度

Part 4 · Harness Engineering

新的应用形态

  • 全新的交互形式和技术栈

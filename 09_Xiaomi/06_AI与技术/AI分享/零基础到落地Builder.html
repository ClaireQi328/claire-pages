核心观点
3 句话记住 Builder
❶ Builder 痴迷于"车"，不痴迷于"工厂"
聚焦产品本身，而非生产流程。你的价值不是优化流水线，而是造出值得造的车。
❷ AI 承担 99% 汗水，人负责 1% 灵感
执行被自动化，判断力成为核心。你的竞争力是"做什么"和"不做什么"。
❸ 学习通过构建发生，不通过学习发生
Day1 装工具 → Week1 构建真实项目 → Month3 超越"资深"。

为什么是现在
"信息搬运者"的黄昏

  • 把需求搬到 PRD → 搬到排期会 → 搬到站会 → 搬到邮件
  • 全程没有创造增量价值
  • 一周结束很忙，但没有产出任何用户可见的东西
✓ Builder 的典型一周
  • 用 Claude Code 写了一个功能原型
  • 跑了一轮用户测试，基于数据调整策略
  • 发布了一个可用的 MVP
  • 一周结束时，有东西可以展示
影响力来自你造出了什么，不来自你管着什么。

能力模型
必修三项：入门门槛
🧱
Context Engineering
上下文工程 · Builder 第一能力
把业务逻辑、约束、验收标准打包成 AI 能消费的结构化上下文。AI 理解不了你的指令 → 说明需求本身没理清。
❌ 模糊上下文
"帮我做个数据看板"
✅ 好的上下文
"做一个SU7 vs Model 3 vs 极氪007 三款竞品对比看板：
• 4 个 KPI：售价 / 续航 / 加速 / 销量
• 深色背景 + 小米橙强调色
• 底部折线图展示近 6 月销量趋势
• 用 HTML + Chart.js，双击可打开"
好上下文的 4 个关键要素
① 角色 "你是一个汽车行业数据分析师"
② 约束 "纯前端，不用后端，单文件 HTML"
③ 数据 "@sales_data.csv 是真实销量数据"
④ 验收 "双击 HTML 能在浏览器跑起来"
🔨
Vibe Coding
AI 辅助编程 · 自然语言→代码
  • Claude Code — 终端级，复杂项目
  • Cursor — AI 原生编辑器
  • Lovable/v0 — 零代码快速验证
🎯
Vibe Checking
质量验证 · 跑起来 ≠ 做对了
  • 功能验证 — 边界测试、异常流
  • 架构判断 — 可维护性、安全
  • 风险兜底 — 知道什么必须交给工程师
上下文写得好 → Vibe Coding 产出精准 → Vibe Checking 更轻松 · Context Engineering 是起点，决定终点。

避坑指南

❌
"Builder 就是不需要产品经理了"
→ Builder 是产品经理的升级版，不是替代品。
❌
"工具会就行，不需要产品感觉"
→ 产品判断力才是真正的护城河——第四阶段的重点。
❌
"非技术背景是劣势"
→ 反而是优势。"把需求讲清楚让 AI 执行"恰恰是 PM 的基本功。

适合人群硬核开发者PM / 日常开发

实战流程
Builder 用 Claude Code 做原型：6 步闭环
  1. 打开 VS Code，进入项目文件夹
    没有项目？新建空文件夹就行
  2. 打开 Claude 面板，用自然语言描述需求
    "帮我做一个竞品信息看板，展示3个竞品核心参数对比"
  3. Claude 生成代码，Diff 预览
    逐个确认或一键全部接受
  4. 在终端运行看效果
    Claude 告诉你怎么跑，复制粘贴命令即可
  5. 不满意？继续对话迭代
    "把颜色换成小米橙""加个筛选功能""能导出Excel吗"
  6. 满意！30分钟-2小时，原型搞定
    可直接在评审会演示

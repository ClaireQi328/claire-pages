一、五层架构全景：从主数据到智能派活
图 1 · 品类主数据搭建五层架构全景 —— 从左到右依次推进，每层标注了由 CLI / Skill / Agent / API / Future 哪种角色承担。第③层"任务路由和通知"是本方案的核心（规则引擎在这里工作），第④层对外暴露为 MCP Tool。
🎯
这张图的读法：从左往右是"从地基到屋顶"的建造顺序。第 ① 层先把品类、人员、MVPS 这些主数据建好（CLI 拉取 + Skill 管理）；第 ② 层有了数据就能做供应商匹配（Skill 推荐 + Agent 自动回写）；第 ③ 层数据齐了就能自动分活（Agent 监听事件 + Skill 统计）；第 ④ 层把能力打包成标准接口供外部调（MCP/API）；第 ⑤ 层是未来的策略智能化方向。
说人话就像盖楼：先打地基（主数据），再砌墙（供应商匹配），再装水电（自动分活），最后通电接网（MCP 对外）。每层都有专人干不同的事——有的管搬运（CLI）、有的大工（Agent）、有的管工具箱（Skill）、有的管总开关（API）。
1.1 四种角色的分工
角色像啥在哪层出现最多典型动作
CLI搬运工——定时搬货第①层（主数据）ep category sync / ep route batch-dispatch
Skill工具箱里的工具全层都有classify / route-match / supplier-recommend
Agent大工——自己干活不用催第②③层（核心）监听事件→串Skill→自动分活+飞书通知
API总开关——对外的门面第④层（MCP）MCP Tool 封装 / 标准化接口

二、规则引擎 —— 派活的"大脑"（在第③层工作）
✅
一句话：规则引擎 = 写死的《分工表》。它只回答一个问题——"这条任务该派给哪个采购员？" 同样情况永远同样答案，不偷懒不记仇。它是第③层"任务路由和通知"的核心组件，被 Agent 通过 route-match 这个 Skill 来调用。
2.1 一条规则长啥样
{ "id":"R-002", "priority":1, "conditions":[# AND 全满足才命中 {field:"module",op:"in",value:[SOR,BDL]}, {field:"category",op:"eq",value:"动力总成"}, {field:"mvps",op:"in",value:[V1]}, {field:"amount",op:"between",value:[100万,500万]} ], "action":{assign:"张明"}, "overflow":{when:"张明.load>80%",fallback:"王磊"} }
说人话就是一句话："要是 SOR 且是动力总成且 V1 且金额 100~500 万，就派张明；忙就转王磊。"
2.2 匹配方式
优先级顺序扫描：从上往下比，第一条全中就用它。没中就报"派不出"，交给 Agent 兜底。初期够用，上百条后再升级索引。
2.3 必须可解释
📝
每次派活都留记录："任务 X → 派给张明，因为命中 R-002（动力总成 ∩ V1 ∩ 金额区间）"。让人敢不改、能兜底。
2.4 它不是 Agent
引擎只管"算派给谁"——没记忆、不学习、不主动触发、不兜底。Agent 才是带着这条政策去干活、还会改进政策的人。引擎是被 Agent 调用的一个工具。
✅ 引擎做
IF→THEN 匹配 · 可解释 · 确定性输出
❌ 引擎不做
触发 · 记忆 · 学习 · 兜底 · 算负载 · 碰界面
🤖 Agent 补上
事件触发 · 反馈学习 · 异常兜底 · 通知推送

第四步+ ⑤ 策略基座feedback-learn成熟准确率持续提升，改派越来越少有，会自我改进

01
01背景与目标
先对齐"为什么做、做到哪"，再进入模块拆解。
目标：把 SOR / BDL / EWO / CWO / SWO / 框架协议 等任务的"人工派活"改为"三级自动路由"——任务分类（品类）→ 匹配 MVPS 级别 → 落到采购员，派得准且派得清（可解释、可改派）。
痛点：派活靠人记人派，慢、易错、人走断档；规则散在脑子里或写死代码，改一次发一次版。
1.1 范围（In / Out）
做（In Scope）不做（Out Scope）
品类主数据 + 人员树 + MVPS 映射维护供应商准入评估模型本身（仅对接映射池）
路由规则配置 + 自动/手动派活 + 批量RFQ 寻源算法（只消费定点结果回写）
6 模块任务列表与路由状态展示飞书/企微卡片样式设计（仅定协议与字段）
规则引擎 + Agent 事件驱动 + CLI资源池策略 AI 决策（列为 Future）
📌
五层架构 ↔ 模块映射（开发对照）：① 品类主数据搭建 → M1；② 定点供应商匹配 → M4；③ 任务路由和通知 → M2/M3/M5；④ 封装成 MCP → M6；⑤ 资源池与策略基座 → Future。

开发建表 / 建索引依据。所有实体带 version（版本追溯）与 updatedAt。
3.1 主数据实体
实体关键字段说明
Category 品类id, name, level(L1/L2), parentId, mvpsMapping{ V1/2/3 → buyerId }, status(active/archived), effectiveDate, version品类树节点；mvpsMapping 为路由核心

3.2 业务实体
实体关键字段说明
Task 任务id, module(SOR/BDL/EWO/CWO/SWO/框架协议), categoryId, mvps, amount, owner(projectId), assigneeId, routeStatus(assigned/failed/pending), routeReason, createdAt, notifiedAtM3；routeReason 必填（可解释）
RouteLog 路由日志taskId, ruleId, matchedConditions, assignedBuyer, overflowUsed, costMs, createdAt每次派活落一条，供 explain/复盘
Exception 异常taskId, type(no_rule/all_overflow/repeat_fix), suggestedBuyer, status(open/done), handledByM5 异常队列

开发必须：每条 Task 派活后写 RouteLog（命中规则 + 原因），这是「可解释」与「准确率统计」的数据基础，缺一不可。
04
04规则引擎（派活算法）
M2/M3 共用。引擎是纯函数：输入 Task，输出 assigneeId + reason。无状态、可解释。
4.1 规则结构（DB / JSON 一致）
{ "id":"R-002", "priority":1, "enabled":true, "conditions":[ # AND：全满足才命中 {field:"module", op:"in", value:[SOR,BDL]}, {field:"category",op:"eq", value:"动力总成"}, {field:"mvps", op:"in", value:[V1]}, {field:"amount", op:"between", value:[1000000,5000000]} ], "action" :{assign:"张明"}, "overflow":{when:"buyer.loadRate>80", fallback:"王磊"} }
4.2 匹配算法（伪代码，后端照写）
# route-match(task) -> {assigneeId, ruleId, reason} rules = db.rules.filter(enabled).sort(priority ASC) for r in rules: if all( match(task[c.field], c.op, c.value) for c in r.conditions ): buyer = r.action.assign if buyer.loadRate > 80 and r.overflow: buyer = r.overflow.fallback # 溢出 return {buyer, r.id, reason=f"命中 {r.id}: {conds}"} return {assigneeId:null, ruleId:null, reason:"无命中规则"} # → 异常队列
4.3 字段与操作符表
field支持 opvalue 示例
modulein / not_in[SOR, BDL]
categoryeq / in动力总成
mvpseq / in[V1, V2]
amountbetween / > / <[1000000, 5000000]
项目(project)eqMX11（任务自带车型项目，来自上游 SOR/BDL，不依赖项目管控模块）
寻源方式(sourcing)eq战略寻源
阶段(stage)eq试制NPI / 量产
紧急度(urgency)eq紧急
区域(region)eq海外
✅
可解释输出格式（routeReason 落地标准）：
"命中 R-002（动力总成 ∩ V1 ∩ 金额区间 100~500万）→ 派张明；负载62%未溢出"
异常时："无命中规则：category=内饰,mvps=V3 无对应规则 → 建议李娜"

05供应商匹配回写事件协议（M4 专章）
M4 的核心不是页面，而是「事件驱动回写」。本章定义 RFQ 定点完成 → 自动写映射池的协议，供后端与 SRM/RFQ 系统对接。
5.1 事件定义
项定义
事件名rfq.quotation_awarded（RFQ 定点/授标完成）
来源SRM / RFQ 系统（ESB / Webhook 推送）
监听方Agent（M4 订阅）；亦支持人工在品类详情点「刷新供应商」触发 supplier-recommend
触发结果写 SupplierMap，更新品类合格供应商池
5.2 事件载荷（Payload 字段）
{ "event": "rfq.quotation_awarded", "categoryId": "动力总成", # 对应 Category.id "rfqId": "RFQ-8821", "suppliers": ["sup-A", "sup-B"], # 定点供应商列表 "awardedAt": "2026-07-14T10:00:00", "source": "RFQ" # RFQ / Manual }
5.3 回写流程
Agent 收到事件 → supplier-recommend 校验品类与供应商匹配 → 写 SupplierMap(categoryId, suppliers) → 更新品类「合格供应商」池 → 失败则进异常（同任务异常机制）。
# 伪代码 on event "rfq.quotation_awarded" (payload): rec = supplier_recommend(payload.categoryId, payload.suppliers) # 校验 if rec.valid: db.SupplierMap.upsert(categoryId=payload.categoryId, suppliers=payload.suppliers, source=payload.source, updatedAt=now()) notify("品类 {categoryId} 合格供应商已更新") else: raise Exception(type="supplier_mismatch", suggested=payload.suppliers)
5.4 定时校验（每日 01:00）
与 M1 的 ep category sync 同源机制：每日 01:00 批量比对 SupplierMap 与 SRM 实际定点结果，差异项告警并标记待人工确认。复用 CLI：ep category sync --source srm --mode incremental。
5.5 人工触发（按需更新）
触发方式动作CLI
品类详情「刷新供应商」按钮调 supplier-recommend 重算并预览—（页面调 Skill）
命令行批量刷新指定品类重跑推荐ep category sync --category 动力总成
🔗
与路由的关系：SupplierMap 是「品类→供应商」映射池；任务路由（M3）派的是「采购员」，二者在不同维度。但同一品类的合格供应商池可辅助 route-match 在 V3 一般件场景下的候选排序。M4 与 M3 共享 Agent 与异常机制。

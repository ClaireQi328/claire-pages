L1 · RFQ 维度
① 统计维度
L1 分析粒度定义
L1 以 报价包（RFQ） 为统计单元，取最高轮次数据，统计该报价包的最高拆解程度。

统计对象：每个 RFQ 作为一个整体参与统计，只看其供应商在最高轮次（最新完成轮次）中达到的最高拆解层级。
典型问题："有多少报价包？""各报价包最高达到了什么拆解程度？"
输出指标：RFQ 总数、供应商总数（去重）、RFQ 最高层级分布。
不拆解维度：不展示单个供应商详情，不展示逐轮变化过程。
② 指标定义
RFQ 总数 & 供应商总数
RFQ_Count = COUNT(DISTINCT RFQ_ID)
Supplier_Count = COUNT(DISTINCT supplier.name) -- 全量去重
报价单元数 = 每个 RFQ 下 各供应商有效轮次数之和（不含退出轮次）。
当前数据：26 个 RFQ，48 家供应商（去重）。
全部报价单元：161 个（所有轮次累计，包含每轮每个供应商的报价记录）。
L2 报价单元（供应商×最高轮次）：69 个（仅取每个供应商的末轮状态，排除已退出供应商）。
③ 指标定义
RFQ 最高层级判定规则
对每个报价包（RFQ），取其所有供应商在最高轮次（最新完成轮次）中的状态，取 rank 最高的场景作为该 RFQ 的"最高拆解程度"：
RFQ_HighestTier(r) = MAX( rank(s.last) ) ∀ s ∈ r, s.last ≠ '退出'
关键前提：仅取各供应商的末轮（最高轮次）数据，不看中间轮次。
rank 映射：未拆解=1 → 自行拆解=2 → BOM结构=3 → BOM+VSM=4

归类逻辑（从高到低判定）：
1. 任一供应商末轮=场景4 → RFQ 归入「BOM+VSM 层级」
2. 无场景4但任一末轮=场景3 → RFQ 归入「BOM结构层级」
3. 无3/4但任一末轮=场景2 → RFQ 归入「自行拆解层级」
4. 否则 → 归入「未拆解层级」
L2 · RFQ + 供应商维度
④ 统计维度
L2 分析粒度定义
L2 以 报价包 × 供应商 为统计单元，取最高轮次（最新完成轮次） 的数据，统计各家供应商的拆解情况。

统计对象：每个 RFQ 下的每家在报价供应商，仅取其在最高轮次（末轮）的状态进行归类统计。
与 L1 的区别：L1 是 RFQ 粒度（只看整体），L2 精确到每家供应商，可以回答"这个报价包里有几家达到了 BOM+VSM"。
典型问题："各家供应商达到了什么拆解程度？""各类占比是多少？"
关键约束：末轮为"退出"的供应商 不参与四场景占比计算，仅在退出数中单独体现。
⑤ 指标定义
四场景全局占比计算
统计前提：每家供应商仅取最高轮次（末轮）的状态参与计算，不累加中间轮次。

将所有报价包内各供应商按末轮状态汇总到全局：
P_k(全局) = Σ C_r(k) ÷ Σ A_r × 100%
其中：C_r(k) = RFQ r 中末轮处于场景 k 的供应商数
A_r = 该 RFQ 活跃供应商总数（不含退出）

约束条件：四个场景占比之和 = 100%（不含退出供应商）。
与 L1 的区别：L1 看的是"有哪些 RFQ 达到了 XX 层级"（RFQ 计数），L2 看的是"有多少供应商处于 XX 场景"（供应商计数），粒度更细。
当前数据：L2 统计基数为 69 个活跃供应商×最高轮次，BOM+VSM 38个(55.1%) | BOM结构 9个(13.0%) | 自行拆解 18个(26.1%) | 未拆解 4个(5.8%)。
⑥ 指标定义
散点图坐标与气泡定义
X 轴 = RFQ.suppliers.length -- 该RFQ的供应商总数
Y 轴 = C_r(场景4) ÷ A_r × 100 -- BOM+VSM 占比(%)
气泡大小 = Σ count_valid_rounds(s) -- 该RFQ总报价单元数
气泡大小含义：供应商数 × 有效轮次数。气泡越大说明该 RFQ 数据量越丰富，分析价值越高。
使用场景：识别"供应商多但 BOM+VSM 占比低"的 RFQ（右下角区域），作为重点攻坚目标。
⑦ 过滤规则
报价包明细表展示规则
默认行为：展示全部 26 个 RFQ，无筛选条件
排序规则：按 RFQ 编号升序（RFQ-00123 → RFQ-00148）。
交互规则：点击任意行 → 自动跳转到页面下方的 L3 检索区并展开该 RFQ 的轮次详情。
数据来源：四场景计数列来自 L2（末轮状态聚合），"改善趋势"列来自 L3（逐轮序列对比首末轮）。
L3 · RFQ + 供应商 + 轮次维度
⑧ 统计维度
L3 分析粒度定义
L3 以 RFQ × 供应商 × 轮次 为最小统计单元。

统计对象：每个供应商在每轮报价中的状态，形成完整时间序列。
典型问题："供应商在哪一轮发生转变？""改善路径是什么？""哪些轮次是关键节点？"
数据形态：数组 ["场景2", "场景3", "场景4"] 表示该供应商从自拆解 → BOM结构 → BOM+VSM 的改善路径。
⑨ 过滤规则
RFQ 搜索框匹配逻辑
匹配字段：RFQ.id 或 RFQ.name 包含输入关键词（模糊匹配）
匹配规则：
· 不区分大小写（输入转为小写后匹配）
· 输入为空或全空格 → 显示全部 26 个 RFQ
· 支持部分匹配（如输入 "保险杠" 可匹配 "前保险杠总成"、"后保险杠总成"）
· 输入时实时过滤（onkeyup 事件触发）
无匹配结果：显示 "未找到匹配的RFQ" 提示。
⑩ 指标定义
改善供应商 & 退出供应商判定
改善供应商 = COUNT(s | rank(s.last) > rank(s.first) AND s.last ≠ '退出')
退出供应商 = COUNT(s | s.last = '退出')
前置条件：仅对轮次数 ≥ 2 的 RFQ 计算改善数。
判定示例：供应商路径 场景2 → 场景3 → 场景4，首轮 rank=2，末轮 rank=4，判定为"已改善"。
数据归属：改善数 / 退出数本质上是 L3 的聚合产物——必须先有逐轮序列，才能判断首轮→末轮的变化。
⑪ 指标定义
供应商改善评价体系（四分类）
基于供应商完整轮次序列的综合判定，优先级从高到低：
IF s.last = '退出' → 🟥 中途退出
IF ∀t, s.rounds[t] = 场景4 → 🟩 全程稳定
IF s.last = 场景4 AND s.first ≠ 场景4 → 🟩 已改善
ELSE → 🟨 未改善
评价逻辑说明："全程稳定"优先于"已改善"，即所有轮次均为场景4的供应商不会被误判为"已改善"。
使用场景：帮助采购员快速识别需要重点跟进的供应商（未改善 / 中途退出）。
⑫ 指标定义
轮次间迁移统计逻辑
扫描每个供应商的相邻轮次对，提取 rank 发生变化的状态转移：
MigrationPair(s) = { (rounds[i-1], rounds[i]) | rounds[i-1] ≠ rounds[i] }
关键规则：
· 同状态不统计（如 场景4→场景4 不产生迁移记录）
· 退出节点前的有效轮次才参与统计
· 柱子颜色 = 目标场景的颜色（绿=场景4, 蓝=场景3, 黄=场景2）
· 迁移次数 = 该 RFQ 内所有供应商发生该模式转换的累计次数

典型迁移模式解读：
· 未拆解 → 自行拆解 = 供应商开始配合拆解工作
· 自行拆解 → BOM结构 = 供应商接受了 SVC BOM 报价方式
· BOM结构 → BOM+VSM = SVC 推广的关键一步，转化路径已打通

🔬 Skill 输出稳定性验证方案

基于双 Skill 交叉验证机制  |  确保 Skill 从试跑到固化的质量可控

一、为什么需要交叉验证？

🔴 单 Skill 验证的局限

• 幻觉风险：AI 可能生成「看起来正确但实际错误」的分析结论

• 提示词漂移：同一 Skill 在不同数据上可能产出不一致的结果

• 过拟合风险：Skill 可能只在特定数据集上表现良好，泛化能力差

• 无人复核：缺少独立第二视角的校验机制

核心思路：用 两个独立的 Skill 对同一数据集进行分析，通过对比结果的一致性来验证输出的可靠性。类似审计中的「交叉复核」或软件测试中的「变异测试」。

五、关键对比维度

交叉验证不是简单的「结果相同」，需要从多个维度系统性对比：

对比维度 具体指标 一致性标准 容差范围
数据准确性 总数、占比、均值等核心统计值 数值完全一致或在容差内 ±1%
结论一致性 分析结论的文字表述 语义一致（允许措辞差异） 语义匹配
异常发现 识别出的异常数据/模式 两个 Skill 发现的异常集高度重叠 ≥80% 重叠
分类准确性 数据分类/标签的结果 同一数据被分入相同类别 ≥95% 一致
趋势判断 上升/下降/平稳的趋势判断 趋势方向一致 方向一致
覆盖率 是否遗漏关键数据/维度 两个 Skill 覆盖的维度集合一致 ≤10% 差异

六、结果判定标准

✅ 通过 — 输出稳定

条件：6 个对比维度中 ≥5 项达标

  • 核心数值偏差 ≤1%
  • 分析结论语义一致
  • 异常发现重叠度 ≥80%

处置：Skill 可进入固化阶段，封装为 MCP

⚠️ 警告 — 需人工复核

条件：6 个对比维度中 3~4 项达标

  • 部分数值存在差异
  • 结论方向一致但细节不同
  • 异常发现部分重叠

处置：人工复核差异点，修复后重新验证

❌ 不通过 — Skill 不稳定

条件：6 个对比维度中 ≤2 项达标

  • 核心数值偏差 >5%
  • 结论矛盾或遗漏关键信息
  • 异常发现重叠度 <50%

处置：回退到 Skill 验证阶段，优化提示词和逻辑

🔄 迭代 — 持续优化

条件：每次验证后记录差异根因

  • 建立 Skill 质量基线
  • 追踪版本间的稳定性变化
  • 形成「验证→修复→再验证」闭环

处置：版本管理，逐步提升稳定性阈值

八、实际案例：RFQ 分析交叉验证

📋 案例背景

对「询报价数据分析」Skill 进行交叉验证，选取 117 个 RFQ、652 条报价数据作为测试集。

对比维度 Skill A：RFQ 分析 Skill B：供应商画像 判定
RFQ 总数 117 117 一致 ✓
报价记录数 652 652 一致 ✓
拆解率 13.2% (86/652) 13.2% (86/652) 一致 ✓
零拆解 RFQ 数 80 (68.4%) 80 (68.4%) 一致 ✓
L4-BOM+VSM 2 个 RFQ 2 个 RFQ 一致 ✓
关键结论 「穿透率极低，需制度保障」 「拆解率与供应商等级无显著关联」 互补 ✓

💡 结论

两个 Skill 在数据层面完全一致，在结论层面互相补充（Skill A 关注整体穿透率，Skill B 关注供应商属性维度）。交叉验证通过，Skill A 可进入固化阶段。

九、落地执行清单

  • ☐ 选定 3~5 个核心 Skill，确定两两交叉组合
  • ☐ 为每个 Skill 准备标准测试数据集（含正常/边界/异常数据）
  • ☐ 定义 6 个对比维度的具体指标和容差标准
  • ☐ 编写自动化对比脚本（Python/SQL），输出一致性报告
  • ☐ 建立 Skill 版本管理机制（提示词变更 = 新版本）
  • ☐ 首轮交叉验证，记录基线数据
  • ☐ 修复不一致项，迭代优化 Skill
  • ☐ 通过验证的 Skill 标记为「可固化」，进入 MCP 封装

🎯 一句话总结

两个 Skill 交叉验证 = 给 AI 的分析结果请了一个「独立审计师」。不一致时不是 Skill 坏了，而是发现了需要优化的地方——这正是 Skill 进化的驱动力。
